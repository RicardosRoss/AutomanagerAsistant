# 玄鉴 V2 阶段 B 小阶运行时设计

- 日期：2026-04-22
- 状态：已完成脑暴，待用户 review
- 作用域：`docs/superpowers/specs/2026-04-21-xuanjian-v2-roadmap-design.md`、`docs/superpowers/specs/2026-04-20-xuanjian-canonical-schema-design.md`、`src/config/xuanjianV2Registry.ts`、`src/services/CultivationService.ts`、`src/handlers/cultivationCommands.ts`
- 目标：让 `realmSubStageId` 从 phase A 的惰性默认值变成真实生效的运行时字段，使小阶进入成长链路与状态展示，但不提前半实现主动战斗

## 一、阶段目标与边界

阶段 B 只做 3 件事：

1. 让 `realmSubStageId` 成为真实生效的运行时字段。
2. 让状态展示稳定输出 `大境·小阶`。
3. 让成长链路在修为或境界变化时自动重算并持久化小阶。

阶段 B 明确不做：

1. 不启用主动战斗求解器。
2. 不启用 `battleLoadout` 的槽位限制。
3. 不启用法门、神通的小阶门槛拦截。
4. 不引入独立的小阶经验条。

阶段 B 采用“持久化 + 统一重算”的口径：

1. `realmSubStageId` 继续保留在 canonical 中。
2. 它不是自由写入值，而是 `realmId + currentPower` 的权威派生缓存。
3. 任何修为或大境变化都必须通过统一解析函数重算。

## 二、运行时职责划分

### 2.1 `xuanjianV2Registry`

负责定义小阶配置与区间规则：

1. `胎息六轮`
2. `练气九层`
3. `筑基三层`
4. 未细分大境的默认小阶定义

它还负责提供：

1. 按 `realmId` 查找该大境的小阶定义。
2. 按 `realmId + currentPower` 解析唯一合法的小阶。
3. 按 `realmSubStageId` 返回展示标签与父大境。

### 2.2 `CultivationStateAdapter`

仍只负责 fresh-start 与 legacy 初始化：

1. 新建或迁移到 canonical 的用户，默认落在 `胎息·玄景`。
2. 不承担复杂的小阶流转逻辑。

### 2.3 `CultivationService`

负责在所有修为或境界变化入口调用统一重算逻辑：

1. 获取修仙状态
2. 专注结算
3. 破境
4. 飞升/重置

它是阶段 B 的主要接线点。

### 2.4 formatter / handler

负责将小阶转换成用户可读文案：

1. `/realm` 及其他状态摘要优先显示 `大境·小阶`
2. 未细分大境仅显示大境名
3. 不提前展示未启用的战斗语义

## 三、小阶数据模型与区间规则

### 3.1 实装范围

阶段 B 只强制细分三个大境：

1. `胎息六轮`
2. `练气九层`
3. `筑基三层`

`紫府 / 金丹 / 元婴` 保留为大境显示，不在阶段 B 内细分完整小阶体系。

### 3.2 小阶命名

`胎息六轮` 使用以下命名：

1. `玄景`
2. `承明`
3. `周行`
4. `青元`
5. `玉京`
6. `灵初`

`练气九层` 使用：

1. `一层`
2. `二层`
3. `三层`
4. `四层`
5. `五层`
6. `六层`
7. `七层`
8. `八层`
9. `九层`

`筑基三层` 使用：

1. `初层`
2. `中层`
3. `后层`

### 3.3 区间口径

小阶完全由 `realmId + currentPower` 推导，遵守以下规则：

1. 使用左闭右开区间，最后一段闭合收尾。
2. 小阶切换不直接产生额外奖励或惩罚。
3. 小阶不改变大境突破阈值。

阶段 B 使用“按当前大境修为包络等分”的策略：

1. `胎息` 的修为包络均分为 6 段。
2. `练气` 均分为 9 段。
3. `筑基` 均分为 3 段。

这样做的原因是：

1. 阶段 B 的目标是让小阶真实生效，而不是一次性完成最终平衡。
2. 等分最稳定，也最容易通过测试锁定。
3. 后续 phase C 如果需要调整强度包络，只需改 registry，不需要重做运行时接口。

### 3.4 条目结构

每个小阶条目至少包含：

1. `id`
2. `parentRealmId`
3. `order`
4. `label`
5. `displayName`
6. `minPowerInclusive`
7. `maxPowerExclusive`
8. `isFallback`

其中：

1. `displayName` 用于直接输出 `胎息·玄景` 这类文案。
2. `isFallback` 只用于未细分大境的保底显示。

## 四、小阶自动流转与持久化策略

### 4.1 统一重算函数

阶段 B 必须引入统一解析函数，概念上等价于：

1. `resolveRealmSubStageId(realmId, currentPower)`
2. `formatRealmSubStageDisplay(state)`

这两个函数的分工固定为：

1. `resolveRealmSubStageId` 只返回唯一合法的小阶 ID。
2. `formatRealmSubStageDisplay` 只负责用户可见显示。

### 4.2 触发时机

小阶重算至少发生在：

1. `getCultivationStatus()`
2. 专注结算入口
3. 破境入口
4. 飞升/重置入口

### 4.3 持久化规则

1. 若重算结果与当前 `realmSubStageId` 不同，则写回 canonical 并保存。
2. 若相同，不额外保存。
3. 所有写回都继续通过 canonical 主状态，不单独创建小阶子文档。

### 4.4 异常收敛

统一解析函数需要承担历史状态收敛责任：

1. 若当前大境没有完整细分表，则返回该大境的默认 fallback 小阶。
2. 若 `currentPower` 低于当前大境下限，则夹到首段。
3. 若 `currentPower` 高于当前大境上限但尚未破境，则夹到末段。
4. 若 `realmSubStageId` 与 `realmId` 父子不一致，以 `realmId + currentPower` 的重算结果为准。

## 五、状态展示与用户可见文案

### 5.1 展示规则

`/realm` 与所有复用修仙状态摘要的入口遵守：

1. 若当前大境已细分，则显示 `大境·小阶`
2. 若当前大境未细分，则仅显示大境名

示例：

1. `胎息·玄景`
2. `胎息·青元`
3. `练气·三层`
4. `筑基·中层`

### 5.2 文案层级

阶段 B 只精细化“当前境界”这一行：

1. `当前境界：胎息·青元`
2. `当前境界：练气·七层`

其余状态仍沿用现有结构：

1. 当前修为
2. 道行
3. 灵石
4. 主修功法
5. 突破提示

### 5.3 兼容原则

1. 不改 `/divination`、`/breakthrough` 的主文案结构。
2. 它们若需要展示当前境界，应复用统一 formatter。
3. 不在阶段 B 提前加入“小阶解锁了什么槽位”的提示。

## 六、测试与验收

### 6.1 单元测试

必须锁定：

1. 胎息不同修为映射到 `玄景 / 承明 / 周行 / 青元 / 玉京 / 灵初`
2. 练气不同修为映射到 `一层` 到 `九层`
3. 筑基不同修为映射到 `初层 / 中层 / 后层`
4. 未细分大境回落到 fallback 小阶

### 6.2 集成测试

必须锁定：

1. 专注结算后，小阶随 `currentPower` 自动变化
2. 破境后切到新大境首个小阶
3. 飞升后回到 `胎息·玄景`
4. `getCultivationStatus()` 可以收敛脏小阶并持久化

### 6.3 handler / bot-facing 测试

必须锁定：

1. `/realm` 展示 `大境·小阶`
2. 未细分大境不展示 fallback 占位名
3. 旧入口复用 formatter 后文案不回归

### 6.4 最小验收标准

阶段 B 完成时必须满足：

1. 小阶不再是静态默认值，而会随修为变化自动更新。
2. 读取旧用户或脏状态时，会被统一收敛到合法小阶。
3. `/realm` 能稳定显示 `大境·小阶`。
4. 阶段 B 开启后，不影响 V1 的修为、破境、灵石、道行主循环。
5. 全量测试继续通过。

## 七、与后续阶段的关系

阶段 B 只负责让“小阶成为真实成长状态”，不负责战斗求解与槽位限制。

后续阶段分工固定为：

1. 阶段 C 才接入主动战斗系统，让小阶进入战斗基线与敌我比较。
2. 阶段 D 才将更完整法门、神通池通过 projection 扩进 runtime。

阶段 B 的价值是为 C 和 D 提供稳定接口，而不是提前半实现它们。
